Checkpointing — Resume from Any Point
💾
How Checkpointing Works
LangGraph SqliteSaver — automatic
  • Every node writes its output to checkpoints/trip_{session_id}.db automatically
  • LangGraph's SqliteSaver is attached at compile time — no extra code inside agents
  • If the process crashes (network drop, power off, error) the .db file preserves all completed nodes
  • On re-launch: resume_session(session_id) loads the last state and continues from the next node
  • HITL gates are checkpointed too — user won't be asked the same question twice on resume
🔄
Resume Flow
checkpointer.py
  • At startup: print list of incomplete sessions (if any)
  • User can type session_id to resume or start new
  • graph.invoke(state, config={"configurable": {"thread_id": session_id}})
  • LangGraph automatically skips already-completed nodes
  • Checkpoint file auto-deleted after Gate 4 (PDF) confirms — clean output
📋
What Gets Saved Per Checkpoint
Full TripState snapshot
  • Full TripState after every node (not just diffs)
  • HITL answers (gate decisions + human_feedback text)
  • Real API responses (flight offers, hotel list, places list)
  • Daily budget targets (no need to re-compute)
  • retry_count per agent (won't retry beyond limit on resume)
⚙
Implementation (main.py)
3 lines to enable
from langgraph.checkpoint.sqlite import SqliteSaver
checkpointer = SqliteSaver.from_conn_string(
  f"checkpoints/trip_{session_id}.db"
)

Retry Logic — 3 Retries per Agent
🔁
Retry Strategy
fallback_handlers.py · with_retry()
  • Each agent wrapped in with_retry(fn, max_retries=3)
  • Attempt 1 → fail → wait 2s → Attempt 2
  • Attempt 2 → fail → wait 4s → Attempt 3 (exponential back-off)
  • Attempt 3 → fail → activate fallback (LLM estimate)
  • Checkpoint saved after each attempt so resume skips completed retries
📊
Retry State Tracking
In TripState
  • retry_count — current attempt number (1, 2, 3)
  • failed_agents[] — agents that exhausted all 3 retries
  • api_status{} — "live" | "fallback" per agent
  • Orchestrator reviews failed_agents before approving
  • If critical agent (weather/transport) failed → warn user in HITL Gate 3
🎯
Orchestrator Retry (Itinerary Quality)
Separate from API retry
  • If review score < 7 → Orchestrator triggers itinerary re-generation
  • Max 3 re-generations (not API retries — LLM quality retries)
  • Each re-gen passes review issues as feedback to Itinerary Agent
  • After 3 attempts → force-approve with warnings noted in PDF
  • Each re-gen attempt checkpointed — can resume if interrupted

Gate 0 — Weather Decision Logic
🌧
Bad Weather Triggers
  • Weather score < 5/10
  • Heavy rain / cyclone / flood warnings
  • Extreme heat (> 42°C) for outdoor destinations
  • Dense fog / visibility alerts for hill stations
  • Off-season with majority of attractions closed
🔀
Alternative Suggestion Logic
  • Same travel_type (beach → beach, hill → hill, city → city)
  • Similar budget range & travel distance
  • Weather score ≥ 7/10 during user's travel dates
  • LLM generates suggestions with reasoning
  • Each alternative shown with weather score + 1-line description
✓
If User Keeps Original
  • Weather warning stored in state as a concern
  • Itinerary agent adjusts plan (indoor activities, flexible days)
  • PDF includes weather advisory section
  • No re-geocoding needed — proceed to Orchestrator
🗺
If User Picks Alternative
  • New destination saved to trip_preferences in state
  • Re-geocode to get new lat/long
  • Re-fetch weather for new destination to confirm score ≥ 7
  • All subsequent agents use new destination
  • Memory Agent notes destination switch for future recall
Gate 2 — Places Preview Logic
📍
What is Shown
  • Must-visit — 3-5 top attractions with description, time needed, entry fee
  • Food spots — 3-4 restaurants with cuisine type and must-try dish
  • Activities — 2-3 experiences matching travel_type (beach, trek, cultural)
  • Hidden gems — 1-2 lesser-known local favourites
  • Source shown: "via Foursquare" or "LLM recommended" (if fallback)
➕
User Adds a Place
  • User types: "I also want to visit Dudhsagar Falls"
  • System validates: is it reachable from destination? (LLM check)
  • If valid → added to must_visit list in state
  • Itinerary Agent will schedule it on an appropriate day
  • Budget Agent accounts for any additional entry fee
✕
User Removes a Place
  • User says: "Remove scuba diving" or picks by number
  • Place removed from the relevant category in state
  • Itinerary Agent will not schedule the removed place
  • If it was the only activity → suggest replacement from hidden gems
✓
After Confirmation
  • Final confirmed place list saved to places_data.confirmed in state
  • Itinerary Agent receives only the confirmed list
  • PDF includes a "Places You'll Visit" section with map-ready names
  • Memory Agent saves user's place preferences for next trip
Gate 3 — Budget Approval Logic
📊
What Is Shown
  • Full breakdown: transport · hotel · food · activities · misc · TOTAL
  • Stated budget vs estimated total — difference shown in Rs.
  • Per-day target table: Day 1 Rs.2,210 · Day 2 Rs.2,990 · Day 3 Rs.2,340…
  • Budget verdict: "Over by Rs.3,200" or "Comfortably within budget"
  • Savings tips already computed by Budget Agent (shown here)
💸
Option A — Cheaper Places (same destination)
  • LLM identifies the most expensive confirmed places by entry_fee_inr
  • Suggests free or low-cost alternatives with similar experience
  • e.g. "Replace Spice Plantation Tour Rs.1,800 → Mapusa Market (free)"
  • e.g. "Replace Scuba Diving Rs.2,800 → Snorkelling at Calangute Rs.400"
  • User selects which swaps to accept → places_confirmed updated → budget re-run
🚌
Option B — Cheaper Transport (same location)
  • Transport Agent re-queries all available modes ranked by cost
  • Shows each option: mode · cost saving · extra travel time
  • e.g. "✈ Flight Rs.8,000 (current)" vs "🚂 Train Rs.3,800 — saves Rs.4,200, +8h"
  • e.g. "🚌 Bus Rs.1,200 — saves Rs.6,800, +14h"
  • User picks mode → transport_data updated → budget re-run
♻
Re-run Loop (max 3 rounds)
  • After user picks Option A/B/C → Budget Agent re-runs with updated choices
  • New budget breakdown shown immediately — Gate 3 repeats
  • If still over budget → offer options again
  • After 3 rounds without approval → warn user and let them force-approve
  • All budget revision history saved to state (budget_revisions[])
TripState Schema
class TripState(TypedDict):

  # ── Session + Checkpointing + Logging ────────────────────────────────
  session_id:             str    # uuid4 — used as LangGraph thread_id for SqliteSaver
  checkpoint_path:        str    # "checkpoints/trip_{session_id}.db"
  log_path:               str    # "logs/session_{session_id}.log"
  last_completed_node:    str    # name of last successfully finished node
  cache_hits:             dict   # {geocode: bool, weather: bool, places: bool}

  # ── Core ──────────────────────────────────────────────────────────────
  user_profile:           dict   # {user_id, name, email}
  trip_preferences:       dict   # {source, destination, start_date, end_date, budget,
                                 #  travelers, travel_type, hotel_pref, food_pref, transport_pref}

  # ── NEW: Geocoding (resolved once, used by all agents) ─────────────────
  coordinates:            dict   # {source_lat, source_lon, dest_lat, dest_lon}

  # ── Memory ────────────────────────────────────────────────────────────
  memory_context:         dict   # {past_trips, preferred_hotel, avg_budget, ...}

  # ── Live API outputs ───────────────────────────────────────────────────
  weather_data:           dict   # {condition, summary, concerns, score, raw_forecast[]}
  weather_advisory:        dict   # {triggered, alternatives[], user_choice,
                                 #  destination_changed: bool}
  transport_data:         dict   # {final_mode, flight_offer{}, train_option{},
                                 #  estimated_cost_per_person, duration_hours}
  hotel_data:             dict   # {recommended{name,id,price_per_night,total},
                                 #  top_3[], amadeus_hotel_id}
  places_data:            dict   # {must_visit[], food_spots[], activities[],
                                 #  hidden_gems[], source: "foursquare"|"llm"}
  places_confirmed:        dict   # {must_visit[], food_spots[], activities[],
                                 #  hidden_gems[], user_added[], user_removed[]}
                                 #  set after Gate 2 (Places Preview HITL)
  budget_summary:         dict   # {breakdown{}, total_estimate, within_budget,
                                 #  daily_pool, base_daily}
  daily_budget_targets:    list   # [{day:1, date:"2025-07-01", day_type:"arrival",
                                 #   weight:0.85, target_inr:2800,
                                 #   min_inr:2660, max_inr:2940}] one per day
  budget_revisions:        list   # history of Gate 3 rounds:
                                 #  [{round:1, action:"cheaper_places",
                                 #    swaps:[], old_total:31200, new_total:27800}]
  budget_gate_round:       int    # current Gate 3 round (max 3)
  itinerary:              dict   # {title, overview, days[], packing_list,
                                 #  emergency_contacts}
  review_status:          dict   # {passed, score, issues[], warnings[]}

  # ── Evaluation + Guardrails ───────────────────────────────────────────
  eval_scores:            dict   # {pacing, variety, preference_alignment, overall}
  guardrail_results:      dict   # {input_valid, budget_conflict, output_valid, warnings[]}

  # ── NEW: API call tracking ─────────────────────────────────────────────
  api_status:             dict   # {weather:"live"|"fallback", flights:"live"|"fallback",
                                 #  trains:"live"|"fallback", hotels:"live"|"fallback",
                                 #  places:"live"|"fallback"}

  # ── Orchestrator + HITL ──────────────────────────────────────────────
  orchestrator_decision:  dict
  hitl_approved:          bool
  hitl_gate:              str    # "gate0"|"gate1"|"gate2"|"gate3"|"gate4"|"gate5"
  human_feedback:         str
  hitl_change_count:      int    # tracks how many times user requested changes
  retry_count:            int    # max 3 retries per agent · reset per agent
  failed_agents:          list   # agents that exhausted all 3 retries (force-approved)
  pdf_status:             dict
  pdf_path:               Optional[str]
  error:                  Optional[str]
5 Production Enhancements

These additions were not in the original assignment but make the system production-grade. Each is small to implement but high in impact.

1. API Response Cache
💾
Why It Matters
  • Free API tiers have daily call limits (OWM: 1,000/day · Foursquare: 1,000/day)
  • During development you'll run the planner 20+ times for the same destination
  • Without cache: waste all quota on identical requests
  • With cache: call API once, reuse for hours/days
⚙
Implementation — cache_store.py
def get(key, ttl_seconds):
  path = f"cache/{key}.json"

  return json.load(path)

def set(key, data):
  json.dump({"ts": now, "data": data}, path)

3. Cost Impact at Gate 2 (Places Preview)
💰
Why It Matters
  • User adds/removes places without knowing the budget impact
  • Each place has entry_fee_inr + local transport cost in state
  • Show impact instantly: "Adding Dudhsagar day trip → +Rs.1,400 on Day 2"
  • Running total shown: "Current daily pool spend: Rs.11,200 / Rs.13,000"
  • Helps user make informed choices — not just picking favourite names
⚙
Implementation
  • Budget Agent already outputs daily_budget_targets[]
  • Places already have entry_fee_inr + time_needed_hours
  • Cost impact = entry_fee × travelers + estimated local transport
  • No extra LLM call — pure arithmetic in hitl/checkpoints.py

5. Structured Session Log
📋
Why It Matters
  • When something silently fails (API returns wrong data, LLM gives bad output) — you have no visibility without a log
  • Essential for debugging in production and learning what actually happened
  • Teaches structured logging — a core engineering skill
  • Uses Python's standard logging module — no new dependency
📄
Log Format — logs/session_{id}.log
[14:23:01] INFO weather_agent START
[14:23:01] INFO cache HIT weather_Goa_2025-07-01
[14:23:01] INFO weather_agent DONE score=7 (0.1s)
[14:23:02] INFO transport_agent START
[14:23:03] WARN Amadeus API FAIL retry=1/3
[14:23:05] WARN Amadeus API FAIL retry=2/3
[14:23:07] WARN Amadeus API FAIL retry=3/3
[14:23:07] WARN transport_agent FALLBACK activated
[14:23:07] INFO transport_agent DONE source=llm (5.2s)
⚙
Implementation
  • Single logger initialised in main.py with session_id in filename
  • Each agent calls logger.info() / logger.warning() at start, end, retry, fallback
  • Log level configurable via .env: LOG_LEVEL=INFO
  • Log file auto-closed and archived after Gate 4 PDF confirm
Setup Guide — Get All API Keys
Step 1 — OpenWeatherMap (Weather + Geocoding)
  1. Go to openweathermap.org → Sign up (free)
  2. Click your username → My API Keys
  3. Copy the default key (or generate a new one)
  4. Free tier: 1,000 API calls/day — more than enough for development
Step 2 — Amadeus (Flights + Hotels)
  1. Go to developers.amadeus.com → Sign up → Create an App
  2. Copy the API Key and API Secret
  3. Free sandbox uses test data (realistic format, not live prices)
  4. To upgrade to live: complete Amadeus production application (paid)
Step 3 — RapidAPI Indian Railways
  1. Go to rapidapi.com → Search "IRCTC Indian Railways"
  2. Subscribe to the free tier → Copy your RapidAPI Key
  3. Free tier: 100 calls/month → enough for learning & testing
Step 4 — Foursquare Places
  1. Go to foursquare.com/developer → Sign up → Create Project
  2. Copy the API Key from your project dashboard
  3. Free tier: 1,000 calls/day
